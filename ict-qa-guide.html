Database Systems

DDL/DML, Data Redundancy, SQL Queries, File-Processing Limitations

Q1 Hospital patient records in separate Excel files — redundancy, DDL/DML, SQL ▼
(a) Two Problems from Data Redundancy & Inconsistency 2 Marks

1. Wasted Storage & Update Anomalies: When patient data (e.g., address) is stored in both the billing file and the records file, updating one without the other creates conflicting information. A doctor may bill the wrong address while records show the correct one.

2. Poor Decision-Making: If a patient's diagnosis is updated in one file but not another, a doctor reviewing the outdated file may prescribe incorrect treatment — a potentially life-threatening consequence.

(b) DDL vs DML 2 Marks

Q2 Retail store with spreadsheets — file-processing limitations, DDL/DML roles, SQL ▼
(a) Two Limitations of File-Processing Systems 2 Marks

1. Data Isolation: Data is scattered across files in different formats, making it difficult to access related information simultaneously (e.g., linking customer records to sales).

2. Lack of Concurrent Access Control: Multiple users accessing and modifying files simultaneously can cause data corruption or loss of updates.

(b) Role of DDL and DML in RDBMS 2 Marks

DDL Role: Used by database administrators to create and modify the structure of the database — defining tables, columns, data types, and constraints using commands like CREATE, ALTER, DROP.

DML Role: Used by users and applications to interact with data stored in the database — inserting, retrieving, updating, and deleting records using SELECT, INSERT, UPDATE, DELETE.

(c) SQL Query — Products priced above 1000 1 Mark
SELECT * FROM Products WHERE Price > 1000;
Q3 College attendance & grades in separate files — redundancy & decision-making ▼
(a) How Redundancy & Inconsistency Affect Decision-Making 2 Marks

Redundancy: When student fee records exist in both the finance and registrar files, management may issue wrong financial aid decisions if one file is out of date — leading to wrong resource allocation.

Inconsistency: If a student's grade is updated in the exam department file but not the academic records file, management may wrongly identify a student as failing and refuse enrollment — damaging student outcomes.

(b) DDL vs DML Commands 2 Marks
DDL commands define structure (CREATE, ALTER, DROP, TRUNCATE). DML commands manipulate data (SELECT, INSERT, UPDATE, DELETE). DDL changes are auto-committed; DML changes can be rolled back.
(c) SQL Query — Students in BBA program 1 Mark
SELECT * FROM Students WHERE Program = 'BBA';
Q4 College student records — redundancy problems, DDL/DML with examples, GPA query ▼
(a) Two Problems — Redundancy & Inconsistency

Redundancy: The same student name and address stored in attendance, results, and billing files wastes storage and forces staff to update the same information in multiple places.

Inconsistency: A student's address updated in billing but not attendance leads to conflicting records — reports and notifications go to different addresses, causing administrative errors.

Q5 Hospital spreadsheets — file-based limitations, CREATE vs INSERT, elderly patients query ▼
(a) Two Limitations of File-Based Systems

1. No Security Enforcement: File systems offer no fine-grained access control — a ward clerk can open the same file containing confidential doctor notes and billing details.

2. Difficulty in Data Sharing: Files created by one department (e.g., Excel by billing) are often incompatible with files used by another (e.g., CSV by pharmacy), making integration complex and error-prone.

Virtualization & Cybersecurity

VMs · Hypervisors · CIA Triad · Security Controls

Q1–Q5 Virtualization concepts, hypervisors, and CIA Triad with examples ▼
What is Virtualization?

Virtualization is the process of creating virtual (software-based) versions of physical resources such as servers, storage, or operating systems using a hypervisor.

A single physical server runs a hypervisor (e.g., VMware or VirtualBox) which creates multiple Virtual Machines (VMs). Each VM has its own OS — one runs Windows 10, another Ubuntu, another macOS — all simultaneously, sharing the server's CPU and RAM.
Role of a Hypervisor

A hypervisor (also called a Virtual Machine Monitor) is software that sits between the hardware and virtual machines. It allocates CPU, RAM, and storage to each VM, ensures VMs are isolated from each other, and manages resource sharing efficiently.

TypeDescriptionExample
Type 1 (Bare Metal)Runs directly on hardware — no host OS neededVMware ESXi, Microsoft Hyper-V
Type 2 (Hosted)Runs on top of an existing OSVirtualBox, VMware Workstation
Benefits of Virtualization
  • Cost Reduction: One powerful server replaces 10+ physical machines — saves electricity, cooling, and maintenance costs
  • Hardware Utilization: Servers often idle at 10–15% without virtualization; VMs maximize usage
  • Isolation for Testing: Snapshots allow reverting to a clean state after each test
  • Rapid Provisioning: New VMs are created in minutes instead of days

AI Ethics & Algorithmic Bias

Sources of Bias · Requirement Analysis Prevention Strategies

Core Principle: If historical data encodes discrimination, the AI will automate that discrimination at scale. Prevention must begin at the Requirements Stage, not after deployment.
Q1 Bank AI rejects loan applicants from certain neighborhoods — bias & prevention ▼
1. How Bias Developed 2 Marks

The AI was trained on historical loan data where applicants from certain postcodes had higher default rates — often due to socioeconomic factors (unemployment, poor infrastructure), not individual creditworthiness.

The model learned to associate location with risk, discriminating against entire communities rather than evaluating individual financial behavior. This is called Historical Bias — the data reflected past inequality, and the AI amplified it.

2. How Requirement Analysis Can Reduce This Bias 3 Marks
RA ActivityHow It Prevents Bias
Define Fairness CriteriaExplicitly require that decisions must not correlate with zip code, race, or neighbourhood
Stakeholder ConsultationInclude community representatives and ethicists — not just engineers — in requirements gathering
Data Auditing RequirementsSpecify that training data must be reviewed for demographic imbalances before model training begins
Bias Testing RequirementsMandate testing across different demographic groups to verify equal performance and rejection rates
Legal Compliance CheckEnsure requirements align with anti-discrimination laws (e.g., Equal Credit Opportunity Act)
Replace Biased ProxiesReplace location-based features with fair, relevant predictors: debt-to-income ratio, employment history
Q2 University admission AI favors students from certain schools ▼
1. Source of Algorithmic Bias

Historical admission data was skewed toward students from elite schools. The model interpreted school name as a predictor of academic success, ignoring equally capable students from underrepresented institutions.

This is Representation Bias — certain groups (elite school graduates) were overrepresented in training data, making the model biased in their favor.

2. SDLC Requirement Analysis to Address This
  • Bias Audit: Require a demographic breakdown of historical admission data before use
  • Fairness Metrics: Define measurable equality targets (e.g., equal acceptance rates across school types)
  • Feature Selection Rules: Exclude school name as a feature; focus on GPA, essay scores, extracurriculars
  • Diverse Stakeholder Input: Include underrepresented community voices in defining admission criteria
  • Transparency Requirement: System must explain rejection reasons in auditable, non-discriminatory terms
Q3 Company AI for promotions — consistently favors older employees ▼
1. How Algorithmic Bias Occurred

Past promotion records were dominated by senior (older) employees because promotions historically required long tenure. The model incorrectly associated age/seniority with promotion-worthiness.

This is Label Bias — the labels (promoted / not promoted) in training data encoded institutional favoritism toward seniority, causing younger high-performing staff to be systematically disadvantaged.

2. Requirement Analysis Activities to Prevent This
  • Define Non-Discriminatory Criteria: Require promotion decisions be based only on performance scores, project outcomes, and competency assessments — not tenure or age
  • Historical Data Review: Audit past promotion data for age distribution imbalances before using it as training data
  • Counterfactual Fairness Requirement: Mandate that changing only an employee's age should not alter the model's recommendation
  • HR & Legal Stakeholder Review: Include HR and legal teams in requirements to ensure compliance with age discrimination laws
  • Performance Weighting: Require that recent performance metrics be weighted more heavily than years of service
Q5–Q6 AI hiring tool & facial recognition — training data bias & fairness criteria ▼
Q5 — AI Hiring Tool Favoring Specific Universities

Why Bias Occurs: If most successful past hires came from specific universities, the model treats university name as a success predictor. This is Sampling Bias — the historical hiring pool was not diverse enough to represent all capable candidates.

Q6 — Facial Recognition Performing Poorly on Certain Demographics

How Training Data Creates Bias: If the training dataset contained predominantly images of one demographic group (e.g., lighter-skinned individuals), the model performs well on that group and poorly on underrepresented ones. The model has learned features specific to the majority group, making it less accurate — or more likely to misidentify — minority groups.

Fairness Criteria to Recommend Before Development:
1. Training dataset must include balanced representation across all demographic groups
2. Minimum accuracy threshold must be met for each demographic group individually
3. System must be tested by independent third parties on diverse test sets
4. Deployment must be restricted in high-stakes contexts (law enforcement, hiring) until bias audits pass
5. Ongoing monitoring must flag accuracy degradation across demographic cohorts
